# mathlib v4.29.1 → v4.31.0 — breaking-change cookbook

Forward-port notes collected bumping several projects from the `leanprover/lean4:v4.29.1` toolchain
(mathlib pinned to `v4.29.1`) to `v4.31.0`, skipping v4.30.0. A change may have landed in either step;
this isn't bisected. Each entry: **symptom** (the error you see) → **cause** (what changed) → **fix**.
Corrections and additions welcome.

## The one debugging technique that matters

When a tactic fails with `unsolved goals` / `made no progress`, insert **`trace_state`** right before (and
after) it, then build just that one file against the cached deps:

```bash
lake env lean Path/To/File.lean      # fast: compiles one file against prebuilt oleans
```

Read the actual goal v4.31 produced. The overwhelming majority of breaks are a tactic (almost always
`convert`) leaving a goal it used to auto-discharge. Fix it, delete the `trace_state`, rebuild. `lake build`
surfaces errors in dependency order, so fixing one file lets the build reach the next file's break — a
cascade of `build → fix → build` is normal, not a sign something is wrong.

**Two caveats on that loop:**

- **`lake env lean <File>` can false-pass.** It elaborates one file against whatever dependency oleans are
  *on disk* — which during a bump may be stale, or CoW-copied from the old version — so it can miss a real
  error that `lake build` catches (e.g. an argument that went implicit→explicit deep in a proof body). Use
  `lake env lean` for fast inner-loop iteration, but **gate each file on `lake build <Target>.<File>`**
  before calling it done.
- **The cascade reaches files outside your initial broken-list.** A bump brief listing "8 broken files" is
  a starting point, not a boundary: once those compile, `lake build` surfaces the next dependency layer —
  often via downstream *proof bodies*, not signatures. "0 fixes in this file so far" says nothing about the
  next layer; keep building until the whole target is green.

# Part 1 — Behavior changes (no deprecation warning)

These have no renamed name to alias, so mathlib's `@[deprecated]` mechanism can't warn you. You get a hard
error or an unsolved goal with no pointer to the cause. This is the part the changelog and deprecation
warnings don't help with.

## A. `convert … using N` now emits a spurious typeclass-instance goal
`convert` splits an `a < b` (or `a ≤ b`) congruence all the way down to the order *instance*, leaving e.g.
`⊢ Real.instLT = Real.instPreorder.toLT` sitting next to the real goal. A trailing `ring`/`norm_num` then
hits the instance goal and errors.

```lean
-- before
convert this using 1
ring
-- after
convert this using 1 <;> first | rfl | ring   -- rfl kills the instance goal, ring the algebra
```
(Seen on a `convert` over a strict-inequality goal in real analysis.)

*Variant — field instance, not order.* A derivative `convert` (e.g. `HasDerivAt.div`'s quotient rule)
splits to the *field* instance instead (`⊢ Real.instField = …`); the trailing closer for the genuine goal
is then usually `field_simp <;> ring`, not bare `ring`:

```lean
convert hquot using 1 <;> first | rfl | (field_simp <;> ring)
```

## B. `convert … using N` stopped auto-unifying `Ico a (b+1)` with `Icc a b`
It now leaves an explicit interval-reindex goal instead of closing it. Discharge it with an `ext … omega`
(proves any such interval identity without guessing a lemma name):

```lean
convert ih 0 using 1
· simp only [Nat.add_zero]
  rw [show Finset.Ico 1 (N + 2) = Finset.Icc 1 (N + 1) from by
        ext x; simp only [Finset.mem_Ico, Finset.mem_Icc]; omega]
· norm_num [qBin_self]   -- the original numeric goal
```
(Seen in a finset-sum induction.)

## C. `convert … using N` stopped unfolding a defeq definition
It leaves `LHS = ∑ Lagrange.basis …` where the two sides are defeq by unfolding. Close with the unfold:

```lean
convert Lagrange.sum_basis … using 1
simp only [Lagrange.basis, Lagrange.basisDivisor]
```
(Seen in Lagrange-interpolation code.)

## D. `ring` / `ring_nf` now ERROR on "made no progress"
v4.29.1 treated no-progress as a silent no-op; v4.31 makes it a hard error. Usually it means an earlier
tactic (`convert`, `simp`) already changed or closed the goal. Either drop the now-redundant `ring`, or
guard it: `first | rfl | ring`. (The same applies to other tactics that used to no-op silently.)

*Sharpening — `ring` inside a `first | rfl | ring` guard can still succeed **noisily**.* After fixing an
entry-A/K `convert` with `… <;> first | rfl | ring`, the build goes green but each such guard prints
`Try this: ring_nf` / `The ring tactic failed to close the goal`. What's happening: `ring`'s fast path
(`ring1`) fails on the post-`convert` goal and logs the suggestion *before* the full tactic closes it via
`ring_nf`. The goal IS closed (swap the `ring` for `skip` and a real error returns) — only the info message
survives. Use **`ring_nf`** in the guard instead; it closes the same goals with no message:

```lean
convert … using 1 <;> first | rfl | ring_nf   -- ring_nf, not ring → no spurious "Try this"
```

## E. `exponentiation.threshold` (default 256) — large `^` left symbolic
A `^` whose exponent exceeds 256 is no longer evaluated at elaboration. This is a **warning**, not an error,
and usually nothing breaks — only a proof that genuinely *relies* on the evaluation (a `decide`/`norm_num`
over a big power) does. Silence it locally if you want clean output:

```lean
set_option exponentiation.threshold 4096 in
theorem … := …
```
Verify the proof's axiom footprint is unchanged (`#print axioms`) if you're unsure whether it relied on the
evaluation; in practice it usually didn't.

## F. `simpa using h` no longer bridges defeq-but-not-syntactic propositions
`simpa using hpos` where `hpos : 0 < n` and the post-`simp` goal is `1 ≤ n` now ERRORS
(`term hpos has type 0 < n but is expected to have type 1 ≤ n`), even though `0 < n` and `1 ≤ n` are
*definitionally* equal in ℕ. v4.31's `simpa` final check is syntactic, not up-to-defeq. Do the simp, then
close with `exact` (which still checks up to defeq):

```lean
-- before:  Nat.log_eq_of_pow_le_of_lt_pow (by simpa using hpos) …
-- after:   Nat.log_eq_of_pow_le_of_lt_pow (by simp only [pow_zero]; exact hpos) …
```

The same syntactic check also fails when the term and goal differ only in their *head* (not the
proposition): a def-wrapped head, a Pi-operation vs its applied lambda, or a defeq-but-different instance
path. Same remedy — `simp only` to normalize the head, then `exact` / `.congr` (both check up to defeq):

```lean
-- Pi-operation head: `Tendsto.div` gives `Tendsto (f / g)`, goal wants `Tendsto (fun n => f n / g n)`
simp only [MkF]; exact num.div den hpos       -- not  simpa only [MkF] using num.div den hpos
-- def-wrapped head:  IntegrableOn f s μ  vs  Integrable f (μ.restrict s)
exact (hIF2.sub hIG2).abs                      -- not  simpa only [Pi.sub_apply] using …
-- instance path: `.mul_left` yields a Summable on a different (defeq) AddCommMonoid instance
exact (summable.mul_left 4).congr fun n => mul_one_div 4 _
```
Heuristic: if the error is *"term has type X but is expected to have type Y"* with X, Y visibly defeq,
stop adding `simpa` lemmas — `simp only [...]; exact h` (or `.congr`) is the fix.

## G. `rw`'s auto-closing `rfl` is reducible-transparency only — force kernel reductions
After a `rw`, a goal like `(match (1 - 1) with | 0 => … | _+1 => …) = Sum.inl (some ?a)` is left *unsolved*
because `rw`'s closing `rfl` won't reduce `1 - 1 → 0` (`Nat.sub` needs default transparency; v4.31's closer
is reducible-only). Append an explicit `rfl` (default transparency) to fire the reduction and assign the
metavariable:

```lean
-- before:  (by rw [foo, hnp])
-- after:   (by rw [foo, hnp]; rfl)
```

## H. A `simp`/`simpa` that used to reduce a match no longer does
`simpa using hp` where `hp : o.SomeProp (Sum.inl none)` and the goal is `o = 0` now type-mismatches because
the reduction no longer fires under `simp`. Use the explicit bridge lemma (often `Iff.rfl`) instead:

```lean
-- before:  … ; simpa using hp
-- after:   … ; rwa [someProp_inl_none] at hp
```
(Seen with `ONote.FundamentalSequenceProp`; the bridge `fundamentalSequenceProp_inl_none : … ↔ o = 0` exists.)

## I. `simp only` / `dsimp only` (no lemmas) errors now that applications are pre-β-reduced
A bare `simp only at h` whose only job was to β-reduce a redex like `(fun i => c (i+1)) j` inside a
hypothesis now ERRORS `made no progress` — v4.31 β-reduces the application at elaboration, so there's nothing
left to do. **Delete the line**; downstream `omega`/`exact`/`rw` see the already-reduced atoms, which is
what they wanted:

```lean
-- before:  have hih := ih (fun i => c (i + 1)) …
--          simp only at hih          -- now "made no progress"
-- after:   have hih := ih (fun i => c (i + 1)) …   -- already β-reduced
```

This also hits a **bare** `dsimp only` / `simp only` (no `at h`) used to β-reduce a redex before a
`rw`/`exact`. When batch-fixing many sites, prefer `try dsimp only` to deleting: a no-op if v4.31 already
reduced and still reduces if not, so you needn't verify each site by hand.

## J. `grind` / `lia` no longer reason past a multiplication
A `grind` (or `lia`) that used to close a `False` goal from mixed facts including a nonlinear one like
`b * s < k * s` now fails. Peel the linear consequence off by hand, then `omega` (which handles nat
subtraction but not multiplication):

```lean
-- before:  · lia
-- after:   · have : b < k := Nat.lt_of_mul_lt_mul_right hres; omega
```

## K. The general `convert` over-split — close goals order-independently
The most common break (A/B/C are instances of it). When `convert … using N` leaves a mix of a spurious
instance goal *plus* genuine goals in a churn-prone order, don't use positional `·` bullets — they misalign
across versions. Use `<;> first | …` so each closer matches whatever goal it lands on:

```lean
convert Polynomial.coeff_eq_esymm_roots_of_card _ _ <;>
  first | rfl | exact hdeg.symm | exact IsAlgClosed.card_roots_eq_natDegree | (rw [hdeg]; exact Nat.sub_le _ _)
```
Related: under `convert`/`simp`, an `esymm` / `Multiset.map_map` composition no longer auto-fires, and an
`IsSymmetric`-membership `Iff.rfl` (`MvPolynomial.mem_symmetricSubalgebra`) must be `rw`'d explicitly.

## M. `simp [Function.comp]` no longer unfolds an *unapplied* `g ∘ f`
`simpa [Function.comp] using (h.comp (tendsto_add_atTop_nat 1))` type-mismatches: the term is
`Tendsto (g ∘ (· + 1)) …` while the goal is `Tendsto (fun N => g (N + 1)) …`. v4.31 unfolds
`Function.comp` only on an *applied* comp `(g ∘ f) x`, so an unapplied `g ∘ f` — e.g. as the function
argument of `Tendsto` — stays folded. Rewrite with the unapplied equation lemma `Function.comp_def`:

```lean
-- before:  simpa [Function.comp] using this
-- after:   simpa [Function.comp_def] using this   -- f ∘ g = fun x => f (g x)
```
(Seen on every `HasSum.tendsto_sum_nat`-style index shift, `(tendsto_… ).comp (tendsto_add_atTop_nat k)`.)

## P. `convert` / `apply` now surface `_`-placeholder args and undetermined implicits as goals
Distinct from the A/K over-split (those leave spurious *instance* goals from a congruence). Here
`convert lemma _ _ … using 1` — or any `apply`/`convert` of a lemma whose implicit isn't pinned by the
other arguments — leaves the unfilled slots as **metavariable goals**: bare *data* goals next to the real
ones (`⊢ ℝ → ℂ`, `⊢ Measure (Fin n → ℝ)`, `⊢ ℝ`), plus their instance goals (`SFinite ?m`, `CommRing ?m`),
and it **reorders** them. A positional closer aimed at one (`exact 1` for the `⊢ ℝ`) then lands on another
and errors — `numerals are data … expected type is a proposition`, or `type class instance expected` when
an `infer_instance` hits the `ℝ → ℂ` goal.

v4.29.1 inferred those `_` placeholders / undetermined implicits by unifying with the goal; v4.31 leaves
them open. **Fix: pin them at the call site** rather than mopping up with positional tactics:

```lean
-- before:  convert mellin_comp_mul_left _ _ h using 1 ; rotate_left ; … ; exact 1
-- after:   provide the function explicitly — there's no convert metavar at all
have hm := mellin_comp_mul_left (fun u => ((Real.exp (-u) : ℝ) : ℂ)) s h
-- before:  convert L_of_bigO _ _ _ _ using 1 ; exact 1        -- the `{r : ℝ}` implicit leaked
-- after:   apply L_of_bigO (r := 1)                            -- 4 clean Prop goals, no data goals
```
Often the cleanest move is to drop `convert` entirely and rewrite the goal into the lemma's exact shape
(`rw [show … = …, lemma]; exact …`), which sidesteps the leak. (Seen with `mellin_comp_mul_left`,
`measurable_measure_prodMk_left`, `LSeriesSummable_of_sum_norm_bigO_and_nonneg`.)

## S. Stricter instance check — non-synthesizable arg / non-class head now hard-error at the declaration
v4.31 rejects two `instance` shapes that v4.29.1 tolerated. The error is on the **declaration**, not a use
site:

1. **An explicit argument that isn't instance-implicit and can't be inferred** (e.g. a hypothesis like
   `(hV : C ⊧* Ax)`, `(ha : a ≠ M.root)`):
   > `This instance has 1 argument that cannot be inferred using typeclass synthesis … not
   > instance-implicit and appear neither in another instance-implicit argument nor the return type`
2. **A return type that isn't a class** — a `Prop` predicate like `Set.Nonempty` / `Set.Countable`:
   > `The declaration 'pInst' should not be an instance as its return type 'P 0' is not a type class.`

Neither could ever fire in synthesis, so the fix is to **demote the keyword to `def`/`lemma`** — type
signature unchanged, explicit call sites unaffected, behavior-preserving:

```lean
-- before:  instance instFrameClassSound (hV : C ⊧* Ax) : Sound (Hilbert.Minimal Ax) C := ⟨…⟩
-- after:   def      instFrameClassSound (hV : C ⊧* Ax) : Sound (Hilbert.Minimal Ax) C := ⟨…⟩
```
(Hit ~29 decls in one port — the `Sound`/`Complete` instances, `set_nonempty`, `set_countable`,
`buildAxioms.instHasJ*`. Worth one line in the PR description so `instance`→`def` in a "mechanical"
bump isn't surprising.)

# Part 2 — Renames and signature changes that hard-error

These *are* the kind of thing `@[deprecated]` normally warns about, but in each case the alias was missing,
`@[to_additive]` didn't propagate, or an argument changed — so you get a hard `unknown constant` /
`Function expected`, not a friendly warning pointing at the replacement.

## L. set-`diff` lemmas renamed `diff → sdiff`; the additive big-operator one hard-errors
v4.31 renamed the set-difference forms `diff → sdiff`. The old name survives as a `@[deprecated] alias` on
the **multiplicative** lemma — but that alias carries **no `@[to_additive]`**, so the *additive* name is now
an `unknown constant` (a real error, not a deprecation warning):

```lean
-- before:  Finset.sum_eq_sum_diff_singleton_add (Finset.mem_univ i₀) f
-- after:   Finset.sum_eq_sum_sdiff_singleton_add (Finset.mem_univ i₀) f   -- same statement
```
The set-level cousins (`Set.image_diff → image_sdiff`, `Set.mem_diff → mem_sdiff`,
`Set.Infinite.diff → sdiff`) keep their aliases, so they stay *warnings* and don't block. Only the
big-operator form without `@[to_additive]` is a hard error.

## N. `zero_le` made its argument implicit — `zero_le _` / `zero_le x` over-applies it
The canonically-ordered `0 ≤ a` lemma (`_root_.zero_le`, for `ℝ≥0∞` / `ℝ≥0` / etc.) made its argument
**implicit** in v4.31, so `zero_le _` or `zero_le c` now errors `Function expected at zero_le`. Drop the
argument → bare `zero_le`. (`zero_le'` also works but is *itself* deprecated → `zero_le`, so don't reach for
it.) When the type can't be inferred from context, ascribe it:

```lean
-- before:  exact zero_le _      |  le_antisymm h (zero_le c)  |  eq_or_lt_of_le (zero_le d)
-- after:   exact zero_le        |  le_antisymm h zero_le      |  eq_or_lt_of_le (zero_le : (0:ℝ≥0) ≤ d)
```
NB this is a *different* lemma from `Nat.zero_le n`, which keeps its explicit argument — leave
`Nat.zero_le _` alone.

*Variant — qualify to a domain lemma instead of dropping the arg.* When the broken `zero_le a` sits in a
library that has its own `0 ≤ a` (e.g. an arithmetic model's `protected Arithmetic.zero_le`, which keeps
an explicit argument), qualifying to that domain lemma is a clean alternative to going bare:
`rcases Arithmetic.zero_le a with …`. The `protected` keyword forces the namespace, so it never collided
with `_root_.zero_le` to begin with — the qualification is idiomatic, not noise. (Seen in the Foundation
arithmetic-model port.)

## O. `ENNReal.mul_le_mul_left` renamed (and the left/right naming convention flipped)
`ENNReal.mul_le_mul_left (h0 : a ≠ 0) (htop : a ≠ ⊤) : a * b ≤ a * c ↔ b ≤ c` is now an `unknown constant`.
The replacement that cancels the common **left** factor is — confusingly — named **`mul_le_mul_iff_right`**
(the convention flipped; read the *statement*, not the name):

```lean
-- before:  refine (ENNReal.mul_le_mul_left hB0 hBtop).mp ?_
-- after:   refine (ENNReal.mul_le_mul_iff_right hB0 hBtop).mp ?_
```
Companions in `Mathlib/Data/ENNReal/Operations.lean`: `mul_le_mul_iff_left` cancels the *right* factor
(`a*c ≤ b*c ↔ a ≤ b`); `mul_lt_mul_iff_{left,right}` likewise.

## Q. `Int.cast_natAbs` renamed `Nat.cast_natAbs` (moved to the `Nat` namespace)
Same statement (`(↑n.natAbs : α) = |↑n|`), now in the `Nat` namespace. The old `Int.`-qualified name is an
`unknown constant`:

```lean
-- before:  Int.cast_natAbs
-- after:   Nat.cast_natAbs
```

## R. A `@[deprecated] alias` only *warns* — but the canonical name behind it can hard-error twice
v4.31 renamed `primesOverFinset` / `coe_primesOverFinset` / `mem_primesOverFinset_iff` →
`IsDedekindDomain.*` and left `@[deprecated] alias`es. The aliases compile (just warn), so leaving them in
*looks* safe — but the canonical name they forward to differs in two ways that **do** hard-error:

1. **A goal carrying the alias won't `rw`-match a lemma stated with the canonical name** — defeq ≠
   syntactic, the same root cause as F. Close it with `exact (Canonical.lemma …).mpr …` (checks up to
   defeq) or `show <canonical>` first.
2. **The canonical lemma made an argument explicit.** `mem_primesOverFinset_iff` flipped `B` from implicit
   to explicit, so `(mem_primesOverFinset_iff hp0).mpr` now resolves `.mpr` against a term that's *still a
   function* → `Invalid field 'mpr' … does not contain Function.mpr`. Supply the new explicit arg:

```lean
-- before:  (mem_primesOverFinset_iff hp0).mpr …
-- after:   (IsDedekindDomain.mem_primesOverFinset_iff hp0 (𝓞 K)).mpr …
```
**General lesson:** a projection/field error naming `Function.<field>` (here `Function.mpr`) means the term
isn't fully applied — an argument silently went explicit; supply it before projecting.

## T. `DFunLike` field `coe_injective'` → `coe_injective` — the alias doesn't reach the `where`-field
v4.31 renamed the `DFunLike` structure field to `coe_injective` (no prime) and shipped
`@[deprecated (since := "2026-06-04")] alias coe_injective' := coe_injective`. That alias rescues
**term-level references** to the *lemma* `DFunLike.coe_injective'` (they compile, just warn) — but NOT the
**field name in a `where` clause**, because `where` / anonymous-constructor field names must match the real
declared field and an `alias` is a separate `def`, not a field. So an instance body hard-errors:

> `'coe_injective'' is not a field of structure 'DFunLike'` + `Fields missing: 'coe_injective'`

```lean
instance : FunLike (M₁ →ₛ[L] M₂) M₁ M₂ where
  coe := …
  coe_injective' := fun φ ψ h => …     -- before
  coe_injective  := fun φ ψ h => …     -- after: drop the prime
```
Same lesson as R: a `@[deprecated] alias` covers *uses of a name*, not *structure-field syntax*. (`FunLike`
is `DFunLike` with a non-dependent codomain, so its instances hit this too.)

## Deprecation warnings (the easy ones)

These name their replacement in the warning text — fix opportunistically, no debugging needed:

- `Polynomial.eval_finset_sum` → `Polynomial.eval_finsetSum`
- `push_neg` → `push Not`
